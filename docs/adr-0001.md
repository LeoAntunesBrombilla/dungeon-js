## ADR-001: Tile types as frozen plain objects

**Date:** 2025  
**Status:** Accepted

### Context

The game needs to represent different types of dungeon cells (wall, floor, door,
etc). Each tile type has fixed properties: a display character, walkability,
transparency, and color. These properties should never change at runtime — a wall
is always a wall. The question was how to model these constants in plain JS.

### Options Considered

**Option A — String constants**
```js
const WALL = "wall";
const FLOOR = "floor";
```
Simple, but properties would need a separate lookup table. Accessing
`TILE_PROPERTIES[WALL].glyph` is awkward and disconnected.

**Option B — Class with static instances**
```js
class Tile {
  static WALL = new Tile("#", false, false, "white");
}
```
Adds `new`, a prototype chain, and `instanceof` checks with no real benefit over
plain objects. Overhead without gain.

**Option C — Plain objects with `Object.freeze()`**
```js
export const WALL = Object.freeze({
  glyph: "#",
  walkable: false,
  transparent: false,
  color: "white",
});
```
Simple, no overhead, immutability enforced at runtime by the engine itself.

### Decision

Plain frozen objects exported as named constants from `tile.js`. Each tile type
is a self-contained object with all its properties inline. No classes, no lookup
tables, no indirection.

### Consequences

- ✅ Simple to read and import anywhere
- ✅ Mutations throw `TypeError` in strict mode (ES modules are always strict)
- ✅ No class or prototype overhead
- ✅ Each tile type is a stable reference — identity comparison works (`tile === WALL`)
- ⚠️ `Object.freeze` is shallow — nested objects would still be mutable (not a
  concern here since all values are primitives)
- ⚠️ No static type checking — nothing stops passing a random object where a tile
  is expected
