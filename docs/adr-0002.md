## ADR-002: WorldMap as a factory function with closure

**Date:** 2025  
**Status:** Accepted

### Context

The game needs a dungeon grid — a 2D structure that holds tile data and exposes
operations like `getTile`, `setTile`, and `isWalkable`. The question was how to
structure this module: as a class, a plain object, or a factory function. This
decision directly touches scope, closures, `this` binding, and value vs reference
— core topics from YDKJS.

### Options Considered

**Option A — Plain object literal**
```js
const worldMap = {
  grid: [],
  getTile(x, y) { return this.grid[y][x]; },
};
```
Works for a single map but doesn't scale — you can't create multiple independent
maps from the same definition. The grid is publicly exposed on the object, meaning
any module can mutate it directly. Also relies on `this`, which breaks when methods
are passed as callbacks.

**Option B — Class**
```js
class WorldMap {
  #grid = [];
  constructor(width, height) { ... }
  getTile(x, y) { return this.#grid[y][x]; }
}
```
Private fields (`#grid`) solve the exposure problem. But `this` binding is still
fragile — passing `map.getTile` as a callback loses `this`. Classes in JS are
syntax sugar over prototypes, adding conceptual overhead without real benefit here.
Also contradicts the YDKJS approach of understanding the underlying prototype
system rather than hiding it behind class syntax.

**Option C — Factory function (closure pattern)**
```js
export const createWorldMap = (width, height) => {
  const grid = /* private, closed over */;
  return {
    getTile(x, y) { ... },  // closes over grid directly
    isWalkable(x, y) { ... },
  };
};
```
The grid lives in the closure — it is genuinely inaccessible from outside. Methods
reference `grid` directly without `this`. Multiple independent maps can be created
by calling the factory multiple times. No prototype chain, no `this` traps.

### Decision

Factory function exported as a named export from `worldMap.js`. The `grid` is a
private variable closed over by all returned methods. Methods never use `this` —
they reference closed-over variables directly. The grid is a 2D array structured
as `grid[y][x]` (rows first, columns second).

### Why `grid[y][x]` and not `grid[x][y]`

The outer array represents rows (vertical position, `y`). Each inner array
represents the cells within that row (horizontal position, `x`). This matches how
2D arrays are conventionally laid out in memory and how you'd think about it
visually: "which row, then which column." Accessing `grid[y][x]` reads naturally
as "row y, column x."

The trap to avoid: `new Array(height).fill([])` creates one array and fills every
row slot with a reference to the *same* array. Mutating row 0 would mutate all
rows. `Array.from({ length: height }, () => Array.from(...))` creates a fresh
independent array for each row, because the map function is called anew per slot.

### Consequences

- ✅ Grid is truly private — no external mutation possible
- ✅ No `this` binding issues — methods use closure variables directly
- ✅ Multiple independent map instances possible (`createWorldMap(10,10)` twice
  gives two completely separate grids)
- ✅ Factory pattern is easy to test — just call the function and assert on the
  returned interface
- ⚠️ All methods are recreated per instance (vs prototype where methods are shared)
  — minor memory cost, irrelevant at game scale
- ⚠️ No `instanceof` check possible — you can't ask `map instanceof WorldMap`.
  Use duck typing instead (`typeof map.getTile === "function"`)
